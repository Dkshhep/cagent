# cagent 开发工作规范(Working Norms)

> 记录日期:2026-06-17
> 适用范围:cagent 仓库下的待办事项(Todo)模块开发,并作为后续工作沿用的约定。

这两条规范由用户与开发协作时约定,优先级高于默认工作方式。

## 规范 1:Spec 先行

任何具体编码工作之前,**先完成 Spec / 需求探讨并达成一致,再动手实现**。

落地方式:
- 每个交付先在本目录(`spec/`)产出一份设计 spec 文档。
- spec 经用户确认后,才进入实现计划与编码。
- 流程:`Spec(spec/) → 用户确认 → 实现计划 → 实现 → 复核`。

## 规范 2:测试用例随产出积累

**所有工作产出都必须伴随测试用例的积累**,以测试作为行为的事实来源。

落地方式:
- 每个交付都要新增 / 扩展 `pytest` 测试,覆盖成功路径与失败路径。
- 沿用现有测试风格:`FakeModelClient` 脚本化输出 + `tests/fixtures/` 隔离 + `tmp_path`。
- 测试随功能一起提交,功能没有对应测试视为未完成。

## 备注

- spec 文档统一放在仓库根目录的 `spec/` 下(覆盖工具链的其他默认位置)。
- 每个子项目 / 模块一份独立 spec,命名 `YYYY-MM-DD-<topic>-design.md`。
